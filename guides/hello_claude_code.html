Hello Claude Code

A step-by-step guide to setting up your research workflow using Claude Code and GitHub Codespaces. No prior developer experience required.

Who this is for

Researchers and students who want to use Claude Code to assist with projects but have little or no prior experience with developer tools.

What you'll have

A fully working cloud development environment connected to Claude Code, with your code protected on GitHub and ready for serious research work.

Time to complete

About 1 hour for the full setup. Parts 4 and 5 are optional and can be done later.

Before you start

You will need accounts on the following services:

GitHub

github.com

Where your code and project files will live

Anthropic

claude.ai

To access Claude Code — requires at minimum a Pro plan at $20/month

Supabase

supabase.com

Optional — only needed for Part 5 (data storage)

Note

This guide focuses on running Claude Code inside GitHub Codespaces — a cloud environment. Nothing gets installed on your personal computer. Everything runs in your browser. This is the recommended approach for beginners because it is safer, more consistent, and works on any device.

Part 1

Create your GitHub repository

A repository is where your project lives. Think of it as a permanent, version-tracked filing cabinet for everything related to your work.

1

Create a new repository

Go to github.com and sign in. Click the + icon in the top right corner and select "New repository".

Fill in the form:

  • Repository name — use a short descriptive name with no spaces, e.g. my-research-project
  • Visibility — set to Private if your work is not ready to be public
  • Add a README file — check this box

Click "Create repository".

2

Protect your main branch

Your main branch should be locked so nothing reaches it accidentally.

Go to your repository → Settings → Branches → Add branch protection rule. Fill in:

  • Branch name pattern — type main
  • Enable "Require a pull request before merging"

Click "Create".

Note

This means nobody — including Claude Code — can push changes directly to main. The only way to update main is through a Pull Request that you personally review and approve. This is your most important safety net.

3

Create a .gitignore file

A .gitignore file prevents sensitive files from accidentally reaching GitHub.

In your repository, click "Add file" → "Create new file". Name it exactly .gitignore and paste this content:

gitignore 
# Credentials - never push these
.env

# Common data files
*.csv
*.xlsx
*.dta
*.parquet
*.rds

# LaTeX compilation files
*.aux
*.log
*.synctex.gz
*.fls
*.fdb_latexmk

# Python artifacts
__pycache__/
*.pyc

# R artifacts
.Rhistory
.RData

# OS files
.DS_Store
Thumbs.db

Scroll down and click "Commit changes" → "Commit directly to main".

Note

The most critical entry is .env — this is where private credentials are stored. It must never be uploaded to GitHub. The .gitignore ensures this file is permanently invisible to Git.

4

Create a CLAUDE.md file

CLAUDE.md is a plain text file that Claude Code reads automatically every time you start a session. It is the instruction manual for your project — the rules Claude always follows without you having to repeat them.

Click "Add file" → "Create new file". Name it CLAUDE.md and paste this content, adjusted to your project:

markdown 
# Claude Code Instructions

## Who I am
This is a solo research project. I am the only person working on this repo.

## Git rules — CRITICAL
- NEVER push to main under any circumstances
- NEVER create a pull request to main without me explicitly asking
- Always work on the dev branch
- NEVER commit .env files or any file containing credentials
- Always commit after each meaningful change with a descriptive message

## Folder structure
- Never modify files inside any input/ folder — these are read-only inputs
- Always write outputs to the output/ folder of the current stage
- Never delete files without asking me first

## Coding style
- Use R with tidyverse unless I specify otherwise
- Comment every function explaining what it does
- Use snake_case for all variable and file names

## General behaviour
- Ask before making structural changes to the project
- If unsure about something, stop and ask rather than guessing
- Summarize what you did at the end of each task

Commit this file directly to main.

Note

You can also create a global CLAUDE.md at ~/.claude/CLAUDE.md for preferences that apply across all your projects — things like your coding language preference or how you like Claude to communicate.

Part 2

Launch your Codespace

A Codespace is a cloud computer that GitHub provides for you to work in. It has VS Code, a terminal, and your repository already loaded.

5

Create your Codespace

Go to your repository on GitHub. Click the green "Code" button. Select the "Codespaces" tab. Click "Create codespace on main".

GitHub will take 1–2 minutes to spin up your environment. When it finishes, VS Code opens in your browser with your project files on the left.

Note

Codespaces has a free tier of 120 core-hours per month. A standard 2-core machine gives you 60 hours of free usage per month — more than enough for research work that does not happen every day. Check your usage at github.com/settings/billing.

6

Open the terminal

In VS Code, go to the top menu → Terminal → New Terminal.

A panel opens at the bottom of the screen. This is your terminal — where you will type all the commands in the steps that follow.

7

Create your dev branch

The first thing to do in every new Codespace is switch off main onto your working branch. Type this command and press Enter:

bash 
git checkout -b dev

Note

git checkout is the command to switch branches. The -b flag means "create this branch if it does not exist yet." From this point, all your work stays on dev and main remains untouched until you deliberately merge.

Part 3

Install Claude Code

8

Check Node.js is installed

Claude Code requires Node.js version 18 or higher. Check by typing:

bash 
node --version

If the output shows v18 or higher, move to Step 9. If not, run:

bash 
curl -fsSL https://deb.nodesource.com/setup_20.x | sudo -E bash -
sudo apt-get install -y nodejs

Note

sudo means "run as administrator" — required when installing software on Linux. apt-get is Linux's built-in package manager, similar to an app store for the command line.

9

Install Claude Code

bash 
npm install -g @anthropic-ai/claude-code

Note

npm is Node's package manager. install -g means install globally, making Claude Code available everywhere on this machine. @anthropic-ai/claude-code is the official package name published by Anthropic.

10

Authenticate Claude Code

bash 
claude

This launches Claude Code for the first time. It will open a browser tab asking you to log in with your Anthropic account — the same one linked to your Pro subscription. Approve it, then return to your terminal.

Claude Code will automatically read your CLAUDE.md file and load those instructions. You are now ready to use Claude Code.

Note

Authentication only needs to happen once per Codespace. If you create a new Codespace in the future, you will repeat this step but it takes less than a minute.

Part 4

Add LaTeX for document writing

Optional

Skip this part if you do not need to write academic papers or formal documents.

11

Update the package list

bash 
sudo apt-get update

Note

This refreshes the list of available software — like refreshing a catalog before ordering. Always run this before installing new software to avoid "package not found" errors.

12

Install LaTeX

bash 
sudo apt-get install -y texlive-full

This takes several minutes — it is a large package. You can continue to the next step in a second terminal window while it installs.

Note

To open a second terminal, click the + icon in the terminal panel. This lets you run the LaTeX install in the background while you keep working.

13

Install the LaTeX Workshop extension

In VS Code, click the Extensions icon on the left sidebar (four squares icon). Search for "LaTeX Workshop" by James Yu and click Install.

This gives you a near-Overleaf experience inside your Codespace: PDF preview that updates automatically on save, side-by-side editing, and click-to-source navigation.

Once installed, Claude Code can write and edit your .tex files, compile them, and iterate based on plain English instructions:

prompt example 
Condense the introduction to 3 paragraphs and make the contribution clearer
Part 5

Store your data in the cloud

Optional

Skip this part if your project does not involve data files, or if your data is small enough to keep only in your Codespace during active work.

Your Codespace is temporary. GitHub itself is not designed for large data files — it has a strict 100MB per file limit and is not meant to store CSVs or datasets. A cloud storage service is the right place to keep your data safe between sessions.

ServiceFree storageBest for
Supabase1 GB foreverResearchers who also want a database
Backblaze B210 GB foreverLarge datasets, budget-friendly
Cloudflare R210 GB foreverLarge datasets, no download fees
Google Drive15 GB foreverIf you already use Google Workspace

Recommended: Supabase. Beyond file storage, Supabase includes a full PostgreSQL database under the same free account. Your data files, cleaned datasets, and structured query results can all live in one place.

Note

If your project grows to need data version control — tracking which version of your data matched which version of your code — tools like DVC can connect to any of these services. That is an advanced topic beyond this guide, but good to know it is possible.

14

Create a Supabase account and project

Go to supabase.com and sign up using your GitHub account. Click "New project" and fill in:

  • Project name — use something general like research, since this one project will hold storage buckets for all your future research projects
  • Database password — save this somewhere safe
  • Region — pick the closest one to you geographically

Click "Create new project" and wait 1–2 minutes.

15

Create a storage bucket

In the Supabase dashboard, click "Storage" in the left sidebar → "New bucket".

  • Bucket name — name it after your current research project, e.g. housing-study. Use hyphens instead of spaces.
  • Visibility — set to Private

Click "Create bucket".

Note

Private means only you — authenticated with your credentials — can access the files. Always use Private for research data.

16

Get your access credentials

In the Supabase dashboard, go to Storage settings and enable the S3 Connection toggle. Click "Create new key" and name it storage-access.

Copy all three values immediately — the secret key is only shown once:

  • Endpoint URL — looks like https://xxxx.supabase.co/storage/v1/s3
  • Access Key ID — a long string of characters
  • Secret Access Key — a longer string, like a password
17

Save your credentials in the Codespace

In your Codespace terminal, create a credentials file:

bash 
touch .env

Open the .env file in VS Code and paste your three values:

env 
SUPABASE_ENDPOINT=your_endpoint_url_here
AWS_ACCESS_KEY_ID=your_access_key_here
AWS_SECRET_ACCESS_KEY=your_secret_key_here

Note

This file is already in your .gitignore so Git will never commit it. You will need to recreate it each time you open a new Codespace — it takes about 30 seconds. Never share these values or paste them into a chat.

Once set up, you can upload files to Supabase directly from your Codespace terminal, or let Claude Code handle uploads as part of your pipeline. Your data files live permanently in Supabase while your code lives in GitHub — two separate, complementary stores.

Part 6

How to work with Claude Code day to day

18

Start a session

Every time you open your Codespace to work, start with:

bash 
git pull origin dev
claude

Give Claude a brief orientation at the start of each session — it has no memory between sessions, so a few sentences of context saves a lot of back-and-forth:

prompt template 
We are working on [project name]. I am on the dev branch.
Today's task is [what you want to do]. Read CLAUDE.md before starting.
19

Save your work to GitHub

After Claude Code makes changes you are happy with:

bash 
git push origin dev

Note

This sends your commits from the Codespace up to GitHub. origin means "send to GitHub" and dev specifies which branch. Without this step, your work only exists in the Codespace and could be lost if it is recycled after inactivity.

20

Publish finished work to main

When you are satisfied with what is on dev and ready to publish it:

  1. Go to your repository on GitHub
  2. Click "Compare & pull request" — GitHub shows this automatically when it detects new changes on dev
  3. Review what changed — read through the differences carefully
  4. Click "Merge pull request"

Note

This is your deliberate review checkpoint. You see exactly what Claude Code produced before it reaches your main branch. If anything looks wrong, close the pull request without merging and go back to fix it first.

Part 7

Protecting your work from mistakes

Claude Code is powerful but not infallible. These layers of protection cover the main risks:

RiskProtection
Claude pushes bad code to mainBranch protection — GitHub rejects it server-side
Claude commits credentials.gitignore — those files are invisible to Git
Claude deletes a file you neededGit history — any committed file can be recovered
Codespace gets recycledGitHub — all committed code is always safe

For long autonomous tasks, add these instructions at the start of your session:

prompt 
Do not push to main under any circumstances.
Do not delete any files without asking me first.
Commit after each completed step with a descriptive message.

Advanced — Docker

For very long autonomous runs that take hours, you can add an extra layer using Docker. This restricts exactly which folders Claude can see. It is not needed for most research workflows, but worth knowing about as your projects grow.

Part 8

Using Skills to standardize repeated tasks

Skills are reusable instruction files stored in a skills/ folder in your repository. Each skill is a markdown file that tells Claude Code exactly how to handle a specific task — like a recipe it follows on demand.

A well-written skill always has three parts:

  1. Clarifying questions — Claude asks before touching anything
  2. Confirmation — Claude summarizes the plan and waits for your approval
  3. Execution — only after you confirm does Claude write any code

To invoke a skill, tell Claude Code explicitly which skill to follow:

prompt 
Read skills/add_figure.md and follow it

Note

The filename alone is not enough — Claude Code does not automatically read skill files just because they exist. You must tell it to read the file in your instruction. You can add reminders in your CLAUDE.md such as "When adding a new figure, always read skills/add_figure.md first" so Claude is prompted automatically.

Glossary

Repository

A folder on GitHub storing all files for one project along with their full change history

Branch

A parallel version of your project where you can work freely without affecting main

Main branch

The primary, official version of your project — stable, reviewed, and trusted

Dev branch

Your active working branch where all development happens before review

Commit

A saved snapshot of your project at a specific moment, with a message describing the change

Pull Request

A proposal to merge changes from one branch into another, requiring your review and approval

.gitignore

A file listing paths that Git should never track or upload

CLAUDE.md

An instruction file that Claude Code reads automatically at the start of every session

Terminal

A text interface where you control your computer by typing commands

Codespace

A complete cloud development environment that runs inside your browser

Node.js

A software runtime that Claude Code depends on to function

npm

Node's package manager — a command-line tool for installing developer software

Skill

A markdown instruction file telling Claude Code how to handle a specific repeated task

Docker

A tool that creates an isolated container limiting what Claude Code can access on the machine

Cloud storage

A permanent remote storage system for files, accessible from any machine

Supabase

A cloud platform offering free file storage and a PostgreSQL database under one account

Bucket

A named container inside a Supabase project used to organize files by research project

PostgreSQL database

A powerful database for storing and querying structured data, included free with Supabase

.env

A local file storing private credentials — never committed to GitHub
